Цепочка обработки чата

Дотошно, с примером, разными ветками и конкретными выходами

1
CRM присылает сообщения
внешняя система
Из мессенджеров (WhatsApp, Telegram, Avito и т.д.) через CRM к нам поступают отдельные сообщения. Каждое сообщение — это: текст, автор, время, мессенджер, ID менеджера, ID клиента.
Может приходить батчем (раз в вечер — все сообщения за день) или по одному (в реальном времени). Оба варианта обрабатываются одинаково.
отдельные сообщения
2
Проверка и сохранение
авто
Каждое сообщение проверяем: не пустое ли? Не приходило ли уже раньше?
Дубликат — выкидываем. Новое — сохраняем в базу как есть, с временной меткой.
Идемпотентность: если CRM пришлёт ту же пачку повторно — ничего не сломается, дубли не создадутся.
сообщения в БД
3
Группировка в диалоги
ключевой шаг
Отдельные сообщения собираются в диалоги. Ключ диалога = менеджер + клиент + мессенджер.
Как это работает:
• За день приходит 30 сообщений между менеджером Алексеем и клиентом Иваном в WhatsApp.
• Все 30 сообщений объединяются в один диалог.
• Если завтра будет ещё 10 сообщений от того же Ивана тому же Алексею — они дополнят тот же диалог.
• Диалог «закрывается» только после длительной тишины (нет новых сообщений, например, 24 часа).

Результат: все сообщения диалога склеиваются в хронологический текст с метками «кто сказал» и временем. Это единица оценки — именно этот текст увидит ИИ.
собранный диалог
Пропускаем
Слишком короткий (1–2 сообщения), ботовый, рассылка, системное уведомление
НЕТ
Диалог подходит для оценки?
ДА
в очередь
5
Разделение на два типа проверок
маршрутизация
Не все правила проверяются одинаково. Есть два типа:
Проверяет КОД (по времени)
5a
Проверки по таймстампам
Правила, которые считаются математически — без ИИ, чисто по времени из метаданных сообщений.
Примеры:
• «Ответ менеджера в течение 1 минуты» — берём время сообщения клиента и время следующего сообщения менеджера, считаем разницу
• «Побуждение при молчании клиента 2 / 5 / 10 мин» — ищем паузы в таймстампах, проверяем написал ли менеджер follow-up
• «Побуждение при молчании 30 мин» — аналогично, но с другим порогом

ИИ тут не нужен. Это чистая арифметика: время ответа = timestamp_ответа − timestamp_сообщения.
результат: число (секунды)
✓
Вердикт по времени
Ответил за 45 сек → выполнено
Ответил за 3 мин → не выполнено
Молчание 12 мин, follow-up есть → выполнено
Обоснование не нужно — результат однозначный, основан на фактах (таймстампы).
Проверяет ИИ (по содержанию)
5b
Проверки по смыслу текста
Правила, которые невозможно проверить без понимания текста — нужен ИИ.
Примеры:
• «Запросил номер телефона в первых 3 сообщениях» — нужно понять, что менеджер именно попросил телефон, а не просто упомянул слово «номер»
• «Обоснование преимуществ объекта» — нужно понять, привёл ли менеджер аргументы, а не просто скопировал прайс
• «Предложил визитку / контактные данные» — нужно найти в тексте предложение связаться
• «Грамотная речь (орфография)» — нужно оценить качество текста

ИИ получает весь текст диалога целиком + список правил. Один запрос = один диалог = все правила разом.
текст диалога + правила
6
ИИ читает и оценивает
ИИ-модель
Модель получает весь диалог (каждое сообщение с [message_id]) и по каждому правилу отвечает:
выполнен / не выполнен + evidence_msg_ids (массив ID сообщений) + цитата.
evidence_msg_ids — массив ID сообщений, на которые ИИ опирается.
CRM может подсветить конкретные сообщения. ID верифицируемый (в отличие от цитаты).
Если ИИ вернул несуществующий ID — ответ невалидный → автоматический retry.
все результаты объединяются
7
Формирование трёх типов результатов
парсинг + валидация
Результаты проверок (и временных, и ИИ-шных) собираются вместе. Ответ ИИ парсится из JSON. Валидация:
1. JSON парсится? Нет → retry
2. Все критерии оценены? Пропущены → retry
3. evidence_msg_ids валидны? Несуществующий ID → retry
4. passed = 0 или 1? Другое значение → retry

Максимум 3 попытки. Между попытками — пауза с экспоненциальным ростом (2с, 4с, 6с).
Чеклист
По каждому критерию
Что содержит:
• Название критерия
• Результат: да / нет
• Обоснование (цитата из переписки)

Пример:
«Запрос телефона» → Да
evidence: [msg_004]
«Оставьте ваш номер телефона»

«Время ответа ≤120с» → Да (75 сек)
evidence: [msg_001, msg_002]
← посчитано кодом, ID подставлены автоматически
Теги
Маркеры диалога
Что содержит:
• Название тега
• Значение: есть / нет (1/0)

Пример:
ипотека = 1
evidence: [msg_002, msg_003, msg_004]
(тема растянута на 3 сообщения)

жалоба = 0 evidence: []

Теги — плоский список. evidence = все сообщения, где тег проявляется.
Данные диалога
Извлечённые факты
Что содержит:
• Название поля
• Значение (текст)
• Только если нашли в тексте!

Пример:
ЖК → Северная долина
evidence: [msg_001]

Тип ипотеки → семейная
evidence: [msg_003, msg_004]
(клиент назвал + менеджер подтвердил)

Если не нашли — поле не возвращаем. Каждое поле = value + evidence_msg_ids.
всё вместе
8
Сохранение в базу
БД
Все три блока (чеклист + теги + данные) сохраняются. Диалог получает статус «оценён» и итоговый процент (сколько пунктов чеклиста выполнено из общего числа).
9a
Callback в CRM
Итоговый балл, оценки по чеклисту, теги — отправляем обратно в CRM. Если CRM не ответила — повторяем позже.
При ошибке — вместе с вердиктом отправляются evidence_msg_ids (CRM подсветит сообщения) + полный текст диалога + цитата-обоснование.
9b
Веб-интерфейс
РОП и администратор видят всё в нашем интерфейсе: список чатов, карточка с деталями, фильтры.
Аналитика: средний балл по менеджеру / офису / неделе, самые частые теги, тренды. Экспорт в Excel.
Сквозной пример: один диалог проходит всю цепочку

10:08Менеджер:Принял, перезвоню в течение часа. Меня зовут Алексей, менеджер отдела продаж
Шаг 3 — Группировка
6 сообщений → 1 диалог. Ключ: Алексей + Иван + WhatsApp. Текст склеен в хронологию.
Шаг 5a — Проверка по времени (код)
Критерий «Ответ ≤ 1 мин»:
Клиент написал в 10:02. Менеджер ответил в 10:03. Разница = 1 мин → ВЫПОЛНЕНО

Критерий «Побуждение при молчании 2 мин»:
Максимальная пауза клиента = 2 мин (10:03→10:05). Менеджер не успел написать follow-up → НЕ ПРИМЕНИМО (клиент ответил сам)
Шаг 5b — Проверка по смыслу (ИИ)
Критерий: «Запрос номера телефона в первых 3 сообщениях»
Менеджер должен запросить у клиента номер телефона. Если клиент сам предоставил без запроса — тоже ок.
Шаг 7 — Результат
5 / 6
Чеклист: 5 из 6 критериев выполнено
ипотека
Тег: тип обращения
Северная д.
Данные: ЖК, бюджет 7.2М, ипотека семейная
Обоснование по «Запрос телефона»: «Оставьте ваш номер, я подберу варианты и перезвоню с деталями» — менеджер запросил телефон во 2-м сообщении. Клиент предоставил номер в ответ.
Единица = диалог
CRM шлёт отдельные сообщения, мы группируем в диалоги по ключу менеджер+клиент+мессенджер.
Два типа проверок
Временные (код считает по таймстампам) и смысловые (ИИ читает текст). Разные ветки.
1 диалог = 1 запрос к ИИ
Все смысловые правила проверяются за один проход. ИИ видит полный контекст.
evidence_msg_ids
Каждый результат = массив ID сообщений-доказательств. CRM подсвечивает конкретные сообщения. Retry при невалидных ID.
